# Test Data (D2) + Secrets

> Title: Auto-generate Test Data & Secrets Contract | Created: 2026-07-02 | Updated: 2026-07-03 (Preconditions schema v2 — seed channel, dependent TCs, pre-flight check) | Version: v2

Markdown is the **source of truth** (QC edits it). A Node build script validates it and compiles a JSON fixture that the spec imports. The md is authored; the JSON is a build output (gitignored — `qc-auto-run` rebuilds it before every run).

**ONE living file per UC** — fixed name `<UC-ID>_testdata.md`, NO date, NO version, edited **in-place** (same exception class as `question-backlog`; history lives in git). Generate never overwrites values the QC has filled; Update appends new variables only.

## Data md schema (must be machine-parseable)

`<UC-ID>_testdata.md`:

```
---
UC: <UC-ID>
Environment: <DEV | QA/Staging | UAT>   # which project-config §3 row gives the base URL
Generated by: qc-auto-generate
---
## Accounts            (role names only — credentials come from project-config §4 at runtime; NO passwords here)
| Key | Role (project-config §4) |
|---|---|
| activeOrgUser | Standard User |

## Variables           (per test case)
| TC | Variable | Value | Note |
|---|---|---|---|
| TC_021 | validEmail | user@org.vn | email khớp tài khoản active |
| TC_018 | — | — | (no input) |

## Preconditions       (data-state the environment must provide — one row per required state)
| Key | Required state | Seed channel | TCs | Check (pre-flight) | Confirmed |
|---|---|---|---|---|---|
| suspendedOrgUser | một tài khoản Org User đã bị ngưng hoạt động, email cố định dành riêng cho test | manual | TC_059 | api: GET /admin/users?email={seed} → expect status=SUSPENDED | — |
| twoOrgsWithPatients | hai tổ chức, mỗi tổ chức có ít nhất một bệnh nhân riêng | ui: setup.flows.ts#seedTwoOrgs | TC_037 | none | — |
| expiredResetLink | một liên kết đặt lại mật khẩu đã hết hạn | manual | TC_105 | none | yes — 2026-07-03 — Chris |
```

### Preconditions field rules

- **Key**: lowerCamelCase, unique. Referenced by specs/fixtures and by `qc-auto-run`.
- **Required state**: Vietnamese, self-contained per `qc-writting-rules.md` R1 — a reader must be able to create the state from this cell alone.
- **Seed channel** (the rung chosen per `automation-feasibility.md` §1 — value REQUIRED):
  - `ui:<flow file>#<function>` — seeded automatically by a generated setup flow (L1/L2);
  - `api:<endpoint ref>` — seeded automatically via the documented API (L3);
  - `db:<script path>` — seeded automatically by a reviewed script under `scripts/db-seed/` (L4);
  - `manual` — the USER must create the state on the test environment BEFORE running (L5).
  Automated channels are executed by setup fixtures; they appear here for traceability. `manual` rows are the ones `qc-auto-run` gates on.
- **TCs**: the TC IDs depending on this state. `qc-auto-run` uses this exact list to Block those tests (and only those) when the state is missing.
- **Check (pre-flight)**: how `qc-auto-run` verifies the state exists before the run:
  - `api: <request> → expect <condition>` (uses base URL + auth from project-config/`.env`);
  - `db: <query> → expect <condition>` (uses the configured connection);
  - `ui: <short probe description>` (a cheap read-only probe);
  - `none` — no automated verification possible.
  Declare a check whenever ANY configured channel allows verification — even for `manual` rows (create by hand, verify by API is a valid and preferred combination).
- **Confirmed**: only meaningful for `manual` rows with `Check = none`. The QC writes `yes — <YYYY-MM-DD> — <name>` after creating the state (QC edits the data md; `qc-auto-run` never writes it). The value MUST start with `yes` — any other text ("no", "chưa", notes...) is treated as UNCONFIRMED (the build warns and strips it). Empty/`—` = unconfirmed → `qc-auto-run` must ask the user before running, or Block the dependent TCs in unattended mode.

## Build script contract (`scripts/build-test-data.mjs`)

Run: `node scripts/build-test-data.mjs <UC-ID>` (wire as `npm run build:testdata`). Behavior:
- Parse the three tables from the data md.
- **Validate** (fail loudly with the offending row): required frontmatter present; table headers exact; no duplicate `(TC, Variable)` or `Key`; every `Accounts.Role` is non-empty; `Seed channel` value is one of `ui:*`/`api:*`/`db:*`/`manual`; every `Preconditions.TCs` cell is a non-empty list of `TC_*` IDs; (optional) every variable referenced by the spec has a value here. A `manual` row with `Check = none` and empty `Confirmed` is VALID but emits a build warning (the run will require interactive confirmation).
- Emit `<UC-ID>_testdata.json`: `{ uc, environment, accounts:{key:{role}}, variables:{TC:{var:value}}, preconditions:{key:{state, channel, tcs:[...], check, confirmed}} }`.
- The JSON is a build output — do NOT hand-edit; regenerate from the md.

## Runtime loader (`helpers/test-data.ts`)

- Imports the generated `<UC-ID>_testdata.json`.
- `vars(tcId)` → the TC's variable object.
- `account(key)` → reads `project-config §4` **at runtime**, finds the row whose role matches `accounts[key].role`, returns `{ username, password }`. Credentials live ONLY in project-config; they are never written into the data md or the JSON.
- `baseURL()` → resolves `Environment` against `project-config §3` (REFUSE if it resolves to PROD).
- `preconditions` — the parsed preconditions map, exposed as a property (consumed by `qc-auto-run` pre-flight and by setup fixtures). Its entry type matches the build output: `{ state, channel, tcs[], check, confirmed }`.

## Environment & `.env` policy (single source of truth)

- **URLs and test accounts are NEVER stored in `.env`** — the single source is `project-config` (§3 URLs per `<ENV> - <portal>` row; §4 accounts by role). Storing them twice guarantees drift.
- **Who fills what**: the USER fills `.env` with external-service secrets only (Mailtrap token, API token, DB connection — per-env values use an env-name prefix, e.g. `QA_MAILTRAP_API_TOKEN`; unprefixed = shared). `qc-auto-generate` scaffolds `.env.example` and never writes secrets.
- **Who loads at run time**: `qc-auto-run`. The user names the target env when invoking ("chạy trên DEV/QA/STG" — matching §3 names); the runner resolves URLs + accounts from `project-config`, CONFIRMS them with the user (usernames only — never passwords), then injects `BASE_URL_<PORTAL>` process vars which `playwright.config.ts` maps onto per-portal `projects`. Specs keep relative routes.

## Secrets policy (HARD)

- Passwords are read from `project-config §4` at runtime, by role. They MUST NOT appear in the data md, the JSON fixture, the spec, or the worklog.
- project-config is a committed file holding test-only accounts; if any account is sensitive, the team should move it to a gitignored `.env` and have `account()` read `process.env` instead — the contract (resolve by role) stays the same.
- API tokens and DB connection strings live ONLY in `.env` (gitignored) — never in the data md, the JSON, project-config, or any generated file.
- Never run against a production base URL (loader `baseURL()` refuses PROD; `playwright.config.ts` also refuses `BASE_URL_*` matching prod).
