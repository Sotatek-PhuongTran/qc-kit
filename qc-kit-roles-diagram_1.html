1 Bản đồ 3 role quanh QC-Kit

QC là người vận hành chính. BA và DEV không chạy pipeline nhưng dùng kit theo role của mình qua các file sống (living docs) được kit quy định chỗ đọc/ghi.

Đầu vào — Requirement

👩‍💼 BA dùng kit như thế nào

  • Cung cấp tài liệu nguồnHigh-level docs, UC spec, wireframe, API doc → khai path tại path-registry.md (nhóm A — "Nguồn: BA")
  • Trả lời Sổ câu hỏiGhi trực tiếp câu trả lời vào Question Backlog (sổ UI per-UC + sổ API portal) do qc-qna tạo
  • Chốt xung đột doc ↔ requirement3 loại finding từ qc-api-tc-design: endpoint thiếu doc, doc mâu thuẫn, endpoint mồ côi
  • Theo dõi tiến độĐọc qc-dashboard.md, project-context-master.md, uc-review-report (verdict + gap)
Vận hành chính

🧪 QC vận hành toàn bộ pipeline

  • QC Lead: onboarding cấu hình dự án, xác nhận môi trường chạy, duyệt kế hoạch verify bug, tổng hợp work-log
  • QC member: chạy chuỗi per-UC — audit UC → thiết kế scenario/TC → sinh automation → run → report → bug
  • Nguyên tắc: kit không tự sửa requirement hay code — chỉ phát hiện gap → đẩy BA, phát hiện defect → đẩy DEV; không đoán, không bịa
  • Chi tiết đầy đủ ở mục 2–4 bên dưới ⬇
Đầu ra — Fix & xác nhận

👨‍💻 DEV dùng kit như thế nào

  • Nhận bug đã phân loại root causebug-report: Lỗi BE / Lỗi FE / Lỗi tích hợp, kèm đối chứng UI↔API — biết ngay sửa ở tầng nào
  • Báo fix qua cột "Trạng thái"Set Dev đã fix — chờ verify (hoặc Không tái hiện / Không phải bug) — tín hiệu kích hoạt qc-bug-verify
  • Trả lời findingsĐiền cột "Trả lời của QC/dev" trong crawl-findings / api-findings
  • BE xác nhận baseline APIChỉ sửa field confirmed trong api-baselines/*.json; cung cấp Swagger/OpenAPI/Postman
BA ⟶ QC: tài liệu + câu trả lời trong sổ
QC ⟶ BA: câu hỏi qua qc-qna (đã lọc qua Question Gate + Dedup Gate)
Quy tắc chung: mọi trao đổi đi qua file sống có cột trạng thái — không trao đổi miệng ngoài luồng; câu trả lời chỉ "có hiệu lực" khi QC re-audit / re-run
QC ⟶ DEV: bug + root cause, findings
DEV ⟶ QC: cột Trạng thái bug, câu trả lời findings, confirmed baseline

2 Pipeline chi tiết cho QC — skill nào, input gì, output gì

Mỗi ô = 1 skill (gọi bằng /tên-skill). I = input, O = output, ✅ = các đầu mục QC phải review / giải quyết xong thì gate của bước sau mới mở. Mọi đường dẫn resolve qua path-registry.md, không hard-code.
