# Naming Convention Rules

> All output files generated by agents MUST follow these naming conventions.
> Updated: 2026-07-10 — API extension step 1: API output types, TS/TC ID conventions, new living-document exceptions.
> Updated: 2026-07-10 (b) — FIX: test case IDs do NOT embed the UC-ID (files are per-UC; matches existing kit practice `TC_021` and `--grep` filters). Scenario IDs keep the UC-ID as before.
> Updated: 2026-07-14 — API extension step 2: thêm `api-question-backlog` (sổ câu hỏi NHÁNH API, 1 file/PORTAL, 2 tầng — xem qc-qna API mode). Sổ `question-backlog` per-UC của nhánh UI GIỮ NGUYÊN; từ nay chỉ chứa câu hỏi nhánh UI.

## General Pattern

```
[UC-ID]_[feature-name]_[type]_[date-or-version]_v[N].[ext]
```

### Components

| Component        | Format              | Example            | Notes                                    |
| :--------------- | :------------------ | :----------------- | :--------------------------------------- |
| `UC-ID`          | `UC-XXX`            | `UC-101`           | Use Case ID from requirement document    |
| `feature-name`   | `lowercase-kebab`   | `user-login`       | Short descriptive name, max 3-4 words    |
| `type`           | See table below     | `extracted`        | Indicates the output type                |
| `date`           | `YYYYMMDD`          | `20260324`         | Date the file was created                |
| `version`        | `vN`                | `v1`               | Version number for finalized files       |

### Variant segment — test case files (both branches)

Test case deliverables insert a `_<variant>` segment between the type and the date:

```
[UC-ID]_[feature-name]_testcases_<variant>_<YYYYMMDD>_v[N].(md|xlsx)
[UC-ID]_[feature-name]_api-testcases_<variant>_<YYYYMMDD>_v[N].(md|xlsx)
```

- `<variant>` comes from `project-context-master` §3.0 (e.g. `web-responsive`, `mobile-native`, `rest`). It is ALWAYS present — even on single-variant projects — so per-variant files pair unambiguously.
- The `.md` and `.xlsx` share the **SAME base name** (differ only by extension). The md is the OFFICIAL machine-readable deliverable (not a draft); both formats are versioned and immutable — update mode writes `v[N+1]` for BOTH.
- Example: `UC-101_user-login_testcases_web-responsive_20260716_v1.md` + `UC-101_user-login_testcases_web-responsive_20260716_v1.xlsx`.

## Scenario & Test Case ID Conventions

| ID | Pattern | Owner |
| :-- | :------ | :---- |
| UI scenario | `TS_[UC-ID]_NNN` | qc-func-scenario-design |
| UI test case | `TC_NNN` (per-UC file scope) | qc-func-tc-design |
| API scenario | `TS_API_[UC-ID]_NNN` | qc-api-scenario-design |
| API test case | `TC_API_NNN` (per-UC file scope) | qc-api-tc-design |
| MIX test case (UI↔API consistency; scope Both only) | `TC_MIX_NNN` (per-UC file scope) | qc-api-tc-design |

Spec titles carry the TC ID verbatim — the run/report pipeline groups results by these prefixes (UI / API / MIX).

## Version Control
- Files use `v[N]` version suffix — they are **immutable** once created.
- When creating a new version, increment `N` (v1 → v2 → v3).
- Never delete or overwrite a version.

### Exception — living documents (no date, no version)
- `question-backlog`: exactly ONE file per UC, fixed name `[UC-ID]_[feature-name]_question-backlog.md`, edited **in-place** by `qc-qna` only. Never add a date segment or `v[N]` suffix; history lives in the file's `Update history` block. Chứa câu hỏi NHÁNH UI (từ `qc-uc-read`); nhánh API chỉ ĐỌC để dedup, câu hỏi API nằm ở `api-question-backlog`.
- `api-question-backlog`: exactly ONE file per PORTAL, fixed name `[PORTAL]_api-question-backlog.md` (portal segment từ UC-ID, vd. `ORGUSER`; segment PHẢI thuộc danh sách portal trong project-context-master §2/§5 — không khớp (vd. segment là tên MODULE) → hỏi user MỘT lần và ghi cách map vào đầu file; không có segment → `ALL`), edited **in-place** by `qc-qna` (API mode) only. Never versioned/dated. Two tiers: mục A câu hỏi chung (cột "UC áp dụng") + mục B theo UC — một câu hỏi nghiệp vụ một dòng (dedup theo nội dung); câu trùng với sổ UI per-UC thì THAM CHIẾU, không chép. Question ID: `Q-API-NNN` (file-scoped), cấp bởi `qc-qna`, không đổi/không tái sử dụng.
- `test-results` (execution report): exactly ONE file per UC, fixed name `[UC-ID]_[feature-name]_test-results.md`, edited **in-place** by `qc-execute-test-report` / `qc-bug-verify` (`qc-bug-report` fills Bug IDs into result cells; QC hand-fills manual-TC cells). Never add a date segment or `v[N]` suffix; history lives in the per-run columns.
- `bug-report`: exactly ONE file per UC, fixed name `[UC-ID]_[feature-name]_bug-report.md`, edited **in-place**. Writers: `qc-bug-report` + `qc-bug-verify`. The user/dev updates ONLY the "Trạng thái" column; history lives in each bug's "Lịch sử" table.
- `crawl-findings` / `api-findings`: ONE file per page / per resource (`<portal>_<page>_crawl-findings.md` / `<resource>_api-findings.md`), edited **in-place**; question columns by the owning generate skill, answer column by the user.
- `test-data` / `api-testdata`: ONE file per UC (`[UC-ID]_testdata.md` / `[UC-ID]_api_testdata.md`), edited **in-place**; QC-filled values never overwritten.
