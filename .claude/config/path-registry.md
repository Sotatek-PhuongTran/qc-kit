# Path Registry

> **Mục đích:** File này lưu các đường dẫn file mà các skills cần tìm để đọc files nhằm giảm thiểu việc phải ghi trong các skills và khó khăn khi cần update lại path.
>
> **Cách dùng:** Khi một skill cần đọc hoặc ghi file, nó tra cứu `logical-name` tương ứng trong bảng dưới để lấy đường dẫn thực tế.
>
> **Cách apply vào dự án mới:** Điền giá trị vào cột `Path` cho từng artifact. Giữ nguyên `docs/` làm gốc.
> Các ô còn để `docs/???` là chưa được cấu hình — agent sẽ hỏi user trước khi tiếp tục.
>

## Phân loại dòng & luật cập nhật

Mỗi dòng thuộc MỘT trong 2 nhóm, suy ra từ cột `Nguồn`:

| Nhóm | Nhận diện | Luật cập nhật |
|---|---|---|
| **A — Input dự án** | `Nguồn = BA` (tài liệu do BA/BE/dự án tạo: high-level, requirement, API doc...) | Khai báo lúc onboarding; cập nhật khi dự án đổi cấu trúc thư mục. Đây là các dòng DUY NHẤT onboarding hỏi path |
| **B — Sản phẩm của kit** | `Nguồn = QC / QC lead / Agent` (mọi input/output do skill trong kit tạo) | Path do kit quy định sẵn — KHÔNG hỏi/sửa lúc onboarding. CHỈ sửa khi phát hiện file thực tế nằm KHÁC path đã khai. **File chưa tồn tại trên đĩa (skill chưa chạy) → GIỮ NGUYÊN giá trị mặc định, KHÔNG xóa thành trống** |

Khi nâng version kit: bước merge chỉ cần chép các dòng nhóm A từ bản cũ sang; toàn bộ nhóm B lấy nguyên theo bản kit mới.

## Quy tắc

- **NEVER hard-code** một đường dẫn cụ thể trong SKILL.md. Luôn tham chiếu `logical-name` từ file này.
- **Luôn đọc version mới nhất** — scan thư mục và chọn file có `v<N>` cao nhất.
- **NEVER overwrite** file đã có. Tạo phiên bản mới `v<N+1>`.
- **Luật cập nhật theo nhóm A/B** (xem bảng đầu file): onboarding chỉ hỏi nhóm A; nhóm B chỉ sửa khi path thực tế khác; file nhóm B chưa tồn tại → giữ mặc định, không xóa trống.
- **Ngoại lệ versioning:**
  - `question-backlog` — 1 file cố định/UC (`<UC-ID>_<feature>_question-backlog.md`, nhánh UI), edit **in-place** bởi `qc-qna`; KHÔNG ngày, KHÔNG version, lịch sử nằm trong khối `Update history` của file;
  - `api-question-backlog` — 1 file cố định/PORTAL (`<PORTAL>_api-question-backlog.md`, nhánh API, 2 tầng: mục A chung + mục B theo UC), edit **in-place** bởi `qc-qna` (API mode); KHÔNG ngày, KHÔNG version, lịch sử nằm trong khối `Update history` của file;
  - `crawl-findings` — 1 file cố định/page (`<portal>_<page>_crawl-findings.md`), edit **in-place**; KHÔNG ngày, KHÔNG version; cột hỏi do `qc-func-auto-generate` ghi, cột "Trả lời của QC/dev" do người dùng điền;
  - `api-findings` — 1 file cố định/resource (`<resource>_api-findings.md`), edit **in-place**; cùng luật với `crawl-findings`; cột hỏi do `qc-api-auto-generate` ghi;
  - `test-data` / `api-testdata` — 1 file cố định/UC (`<UC-ID>_testdata.md` / `<UC-ID>_api_testdata.md`), edit **in-place**; KHÔNG ngày, KHÔNG version; giá trị QC đã điền không bao giờ bị ghi đè; lịch sử = git;
  - `execution-report` — 1 file cố định/UC (`<UC-ID>_<feature>_test-results.md`), edit **in-place**; KHÔNG ngày, KHÔNG version; lịch sử = các cột run trong file + git;
  - `bug-report` — 1 file cố định/UC (`<UC-ID>_<feature>_bug-report.md`), edit **in-place**; KHÔNG ngày, KHÔNG version; cột "Trạng thái" do user update, lịch sử nằm trong bảng "Lịch sử" của từng bug;
  - `qc-api-coverage` — file SỐNG derived view: `qc-api-read` rebuild và ghi đè NGUYÊN FILE ở cuối mỗi lần chạy; KHÔNG ngày, KHÔNG version, KHÔNG sửa tay (nguồn sự thật là cột "Vai trò" + khoá claim trong api-audited §2, endpoint từ bảng binding TC md — muốn đổi thì re-audit UC nguồn);
  - file meta-config (`.claude/config/path-registry.md`) và các file cố định trong `docs/qc-lead/` (`project-config.md`, `project-context-master.md`, `qc-dashboard.md`, `agent-work-log.md`, `agent-work-log.local/worklog_*.jsonl`) được edit **in-place**;
  - file code trong project automation (`*.ts`, `*.json` build output, `playwright.config.ts`, `package.json`) là file code git-tracked, edit in-place kèm stamp nguồn; `api-schemas` và `api-doc-digest` là build output của `parse-api-doc-script` — regenerate, không sửa tay, không version;
  - `reports/summary-latest.md` overwrite in-place, lịch sử chép vào `reports/history/` trước khi ghi đè.
- Nếu path của một artifact **nhóm A** là `docs/???`, agent phải **dừng lại và hỏi user** trước khi tiếp tục.
- Nếu không tìm thấy `logical-name` trong bảng này, agent phải **dừng lại và hỏi user** trước khi tiếp tục.
